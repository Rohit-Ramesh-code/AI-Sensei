## Summary

Phase 4.1 is a targeted gap-closure phase. All five requirements (SNMP-01, SNMP-04, ALRT-02, ANLZ-04, SCHD-01) were previously reported as satisfied but were found to be partial or unsatisfied by a v1.0 milestone integration audit. The gaps are wiring defects — the individual components work in isolation, but the production entry point in `main.py` never connects them.

There are exactly three code changes required, each surgical and bounded:

**Gap 1 (P0, SNMP-01, SCHD-01):** `run_job()` in `main.py` calls `graph.invoke(_build_initial_state())` where `_build_initial_state()` hardcodes `poll_result=None`. The LangGraph analyst node exits immediately on `None`. The fix is to call `SNMPAdapter(host, community).poll()` inside `run_job()`, check the result, and inject it into the initial state before `graph.invoke()`.

**Gap 2 (P0, SNMP-04, ANLZ-04):** `append_poll_result()` is called only for error events (`llm_failure`, `suppressed_alert`, `pipeline_error`). Normal toner `PollResult` data from SNMP is never written to `logs/printer_history.jsonl`. `compute_color_stats()` always reads empty history, meaning the LLM trend-aware path is permanently in cold-start. The fix is to call `append_poll_result(poll_result)` after each successful SNMP poll — before `graph.invoke()`.

**Gap 3 (P1, ALRT-02):** `build_body()` in `agents/communicator.py` has an `llm_reasoning` parameter but no `llm_confidence` parameter. The email body includes the LLM reasoning text but no explicit `Confidence: X%` line. ALRT-02 requires confidence as a distinct structured field. The fix adds `llm_confidence: Optional[float]` to `build_body()`, appends `f"Confidence: {llm_confidence:.0%}"` as a labeled line before the reasoning section, and threads it through `run_communicator()`.

**Primary recommendation:** Apply all three targeted fixes to `main.py` (`run_job()`), `main.py` (`_build_initial_state()` signature or the inline initial state dict), and `agents/communicator.py` (`build_body()` + `run_communicator()`). No new libraries or infrastructure required.

## Architecture Patterns

### Pattern 1: SNMP Poll Before Graph Invoke

**What:** `run_job()` must perform the SNMP poll and inject the result into the initial state dict before calling `graph.invoke()`.

**When to use:** Every scheduled polling cycle. The poll result is the entry data for the LangGraph pipeline.

**Current (broken):**
```python
# main.py run_job() — BROKEN: poll_result starts as None
def run_job() -> None:
    try:
        state = graph.invoke(_build_initial_state())  # poll_result=None — analyst exits immediately
```

**Fixed pattern:**
```python
# main.py run_job() — FIXED: poll SNMP, then inject result
def run_job() -> None:
    try:
        host = os.getenv("SNMP_HOST", "localhost")
        community = os.getenv("SNMP_COMMUNITY", "public")
        snmp = SNMPAdapter(host=host, community=community)
        poll_result = snmp.poll()

        # Persist poll data to JSONL before graph — satisfies SNMP-04
        append_poll_result(poll_result)

        initial_state = _build_initial_state()
        initial_state["poll_result"] = poll_result

        state = graph.invoke(initial_state)
        logger.info(
            "Cycle complete: alert_needed=%s alert_sent=%s suppression=%s",
            state.get("alert_needed"),
            state.get("alert_sent"),
            state.get("suppression_reason"),
        )
    except Exception as exc:
        logger.exception("Pipeline error in polling cycle: %s", exc)
        append_poll_result({
            "event_type": "pipeline_error",
            "printer_host": os.getenv("SNMP_HOST", "unknown"),
            "timestamp": datetime.now(timezone.utc).isoformat(),
            "error": str(exc),
        })
```

**Key constraints from existing code:**
- `SNMPAdapter` import must be added to `main.py` (currently not imported there — `supervisor.py` imports it)
- `_build_initial_state()` returns a fresh dict each call; mutate the returned dict before passing it to `graph.invoke()`
- `append_poll_result(poll_result)` call goes BEFORE `graph.invoke()` — ensures history is available even if graph fails
- The `except` block already calls `append_poll_result` for error events — don't duplicate that call on the happy path

### Pattern 2: JSONL Persistence for Successful Polls

**What:** Every successful SNMP poll writes the `PollResult` dict to `logs/printer_history.jsonl` via `append_poll_result()`.

**Why now:** `compute_color_stats()` in `analyst.py` reads this file to compute velocity. Without poll data in the file, `n < 2` is always true and the LLM trend path is unreachable.

**The serialization note:** `PollResult` is a TypedDict. `QualityFlag` fields are `str` Enums (`.value` is already a plain string). `jsonlines.write()` handles the dict directly — no custom encoder needed. This is a documented project decision from Phase 1 (`[01-01]` in STATE.md).

**Pattern:**
```python
# Correct: call append_poll_result(poll_result) with the live PollResult
append_poll_result(poll_result)  # poll_result is a PollResult TypedDict
```

**Important:** `append_poll_result` has a `log_path` parameter that defaults to `LOG_PATH = Path("logs/printer_history.jsonl")`. The production call should use the default (no argument). Tests must inject a `tmp_path`-based log path to avoid touching real logs.

### Pattern 3: Confidence Score as Standalone Email Field

**What:** `build_body()` adds a labeled `Confidence: X%` line when `llm_confidence` is available, before or alongside the `Analysis:` section.

**Current (partial — ALRT-02 not fully met):**
```python

    flagged_colors: list[dict],

    ...

```

**Fixed pattern — add `llm_confidence` parameter:**
```python

    printer_host: str,
    flagged_colors: list[dict],
    llm_reasoning: "str | None" = None,
    llm_confidence: "float | None" = None,
) -> str:
    ...

            lines.append(f"Confidence: {llm_confidence:.0%}")
        lines.append("Analysis:")
        lines.append(llm_reasoning)
    else:
        lines.append("Note: LLM analysis unavailable — alert based on threshold check only.")
    ...
```

**Thread through `run_communicator()`:**
```python
body = build_body(
    printer_host,
    flagged_colors,
    llm_reasoning=state.get("llm_reasoning"),
    llm_confidence=state.get("llm_confidence"),

```

**Format decision:** `f"{llm_confidence:.0%}"` converts `0.91` → `"91%"`. This is compact and human-readable. The label `"Confidence:"` matches what existing tests check for.

### Anti-Patterns to Avoid

- **Polling inside `_build_initial_state()`:** This would move the SNMP call into a helper that's semantically "build state defaults." Keep it in `run_job()` where side effects belong.
- **Calling `build_graph()` inside `run_job()`:** The existing design compiles the graph once at startup. Never move `build_graph()` inside `run_job()` — that would recompile the LangGraph on every poll cycle.
- **Using `run_pipeline()` in `main.py`:** `run_pipeline()` in `supervisor.py` already calls `SNMPAdapter.poll()` internally. But using it from `main.py` would prevent calling `append_poll_result()` between the poll and the graph invoke. Keep the SNMP call and persistence call explicit in `run_job()`.
- **Opening `logs/printer_history.jsonl` in write mode:** `persistence.py` already uses `mode="a"` (append). Never pass `mode="w"` — that truncates history.
- **Adding `llm_confidence` before `llm_reasoning` in the email body when reasoning is absent:** The confidence line only makes sense alongside LLM analysis. Only show it when `llm_reasoning is not None`.

## Common Pitfalls

### Pitfall 1: Calling `append_poll_result` After `graph.invoke()`
**What goes wrong:** If `append_poll_result(poll_result)` is placed after `graph.invoke()`, and the graph raises an exception, the poll is lost from history — `compute_color_stats()` never sees it.
**Why it happens:** Intuition says "log after the fact." But the JSONL log is the source of truth for trend analysis, not an audit trail.
**How to avoid:** Call `append_poll_result(poll_result)` BEFORE `graph.invoke()`. The happy-path data is written first; the error handler already writes `pipeline_error` events independently.
**Warning signs:** Test passes in isolation but history never accumulates in integration runs.

### Pitfall 2: Forgetting to Import `SNMPAdapter` in `main.py`
**What goes wrong:** `main.py` currently imports `build_graph` from `agents.supervisor` and `append_poll_result` from `adapters.persistence`. It does NOT import `SNMPAdapter`. Adding the poll call without the import causes `NameError`.
**Why it happens:** `SNMPAdapter` is imported in `supervisor.py`, not `main.py`. The refactoring moves the poll out of `run_pipeline()` into `run_job()`.
**How to avoid:** Add `from adapters.snmp_adapter import SNMPAdapter` to `main.py` imports (after `load_dotenv()` call, same group as other project imports).
**Warning signs:** `NameError: name 'SNMPAdapter' is not defined` on first poll cycle.

### Pitfall 3: `_build_initial_state()` Returns `poll_result=None` — Must Patch After Call
**What goes wrong:** `_build_initial_state()` is designed to return all-default values including `poll_result=None`. If the fix modifies `_build_initial_state()` to call `SNMPAdapter` internally, tests that mock `_build_initial_state()` will break and it violates the single-responsibility of a "build defaults" factory.
**Why it happens:** Temptation to put the poll call inside `_build_initial_state()` to minimize changes.
**How to avoid:** Keep `_build_initial_state()` as a pure dict factory. In `run_job()`, call it, then set `["poll_result"]` on the returned dict before invoking the graph.
**Warning signs:** `_build_initial_state()` starts importing `SNMPAdapter` or calling `os.getenv("SNMP_HOST")`.

### Pitfall 4: Confidence Format Mismatch Between `build_body()` and Test Assertions
**What goes wrong:** The existing test `test_build_body_includes_analysis_section` asserts `"Confidence:" in body`. If the implementation uses a different label (e.g., `"LLM Confidence:"` or `"confidence_score:"`) the test fails.
**Why it happens:** Inconsistency between what the test was written to expect and the implementation choice.
**How to avoid:** Use exactly `"Confidence:"` as the label prefix — matches the audit specification and the existing test assertion.
**Warning signs:** Test fails with `AssertionError: Expected 'Confidence:' in body`.

### Pitfall 5: Double-Persisting Poll Result on Error Path
**What goes wrong:** If `append_poll_result(poll_result)` is called before `graph.invoke()` (happy path), AND the `except` block also calls `append_poll_result()` for `pipeline_error`, the poll result is written once before the error and the error event separately after. This is correct. But if the SNMP poll itself raises (before `poll_result` is assigned), the `except` block must not reference `poll_result` (it may be undefined).
**Why it happens:** The `except Exception` block in `run_job()` uses `os.getenv("SNMP_HOST", "unknown")` for the printer host — it doesn't reference `poll_result`. This is the correct pattern. Do not change the except block to reference `poll_result`.
**Warning signs:** `UnboundLocalError: local variable 'poll_result' referenced before assignment` in the except block.

### Pitfall 6: Test Isolation for JSONL Persistence in `run_job()` Tests
**What goes wrong:** New `run_job()` tests that verify `append_poll_result(poll_result)` is called will write to `logs/printer_history.jsonl` if not isolated, polluting the real log file.
**Why it happens:** `append_poll_result` defaults to `LOG_PATH = Path("logs/printer_history.jsonl")`. Unless patched, it writes to the real path.
**How to avoid:** In `test_main.py` tests for the new SNMP-poll behavior, either:
  - `patch("main.append_poll_result")` to mock it and assert it was called with the right argument, OR
  - Pass a `tmp_path`-based log path if testing the actual write (less common for `run_job()` unit tests)
**Warning signs:** Real `logs/printer_history.jsonl` grows during test runs.

### build_body() Fixed Signature

```python
# Source: agents/communicator.py build_body() — Gap 3 fix
def build_body(
    printer_host: str,
    flagged_colors: list[dict],
    llm_reasoning: "str | None" = None,
    llm_confidence: "float | None" = None,
) -> str:
    lines = [f"Printer: {printer_host}", "", "Low toner detected:"]

    for entry in flagged_colors:
        color = entry["color"]
        display_value = entry["display_value"]
        urgency = entry["urgency"]
        lines.append(f"  {color.capitalize()}: {display_value} [{urgency}]")
        lines.append(f"  Recommended action: Order {color} toner")
        lines.append("")

    if llm_reasoning is not None:
        if llm_confidence is not None:
            lines.append(f"Confidence: {llm_confidence:.0%}")
        lines.append("Analysis:")
        lines.append(llm_reasoning)
    else:
        lines.append("Note: LLM analysis unavailable — alert based on threshold check only.")

    return "\n".join(lines)
```

def test_run_job_polls_snmp_and_invokes_graph(monkeypatch):

## Open Questions

1. **What happens when `SNMPAdapter.poll()` raises an exception?**
   - What we know: `SNMPAdapter` is designed to never raise — transport failures return a `PollResult` with `snmp_error` set and `overall_quality_ok=False`. The docstring says "The adapter NEVER raises."
   - What's unclear: Edge cases where the asyncio loop itself throws (OOM, etc.).
   - Recommendation: The existing `except Exception` block in `run_job()` already handles this. No additional guard needed. The try/except covers the poll call too since it will be inside the try block.

2. **Should `append_poll_result()` be called even when `poll_result.snmp_error` is set?**
   - What we know: SNMP-04 says "every poll result (valid or invalid)" is persisted. The word "invalid" explicitly includes error polls.
   - What's unclear: Whether calling `append_poll_result()` for both error polls AND a subsequent `pipeline_error` event double-counts the failure.
   - Recommendation: Yes, persist the poll result unconditionally (including when `snmp_error` is set). The `pipeline_error` event is a separate event type for unhandled exceptions in the pipeline. An SNMP error that returns a valid `PollResult` struct should still be logged — the policy guard reads `overall_quality_ok` to suppress the alert.

3. **Do existing tests for `build_body()` break when the signature changes?**
   - What we know: `build_body()` will gain `llm_confidence: float | None = None` as a keyword argument with a default. All existing callers use keyword arguments (`llm_reasoning=...`). Adding a new optional keyword argument with a default is backward-compatible.
   - What's unclear: None — this is safe. No existing test will break.
   - Recommendation: Use keyword argument with default `None`. The new parameter is opt-in.
